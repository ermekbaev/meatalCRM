## 🟠 Надёжность и масштабируемость

### 6. Пагинация списочных эндпоинтов
**Проблема:** только 3 из 61 роута используют `take/skip` — списки возвращают все строки.

**Инфраструктура:** `lib/pagination.ts` — `getPageParams()`, `paginated()`, константы `DEFAULT_PAGE_SIZE=50`, `MAX_PAGE_SIZE=200`, `SAFETY_TAKE=500`. UI-компонент `components/ui/pagination.tsx`.

- [x] **Полноценная пагинация** (ответ `{ items, total, page, pageSize }` + фронт-контролы + серверный поиск): `clients`, `offers`, `invoices`. Дефолт `pageSize=50`, потолок 200.
- [x] **Защитный потолок `take: SAFETY_TAKE` (500)** без смены формата ответа (остаётся массивом): `tasks`, `requests`, `warehouse`, `catalog`. Эти списки **сознательно не пагинируются классически** — см. ниже.
- [x] Разумный `take` по умолчанию есть везде (для пагинируемых — 50; для остальных — потолок 500).

> **Почему tasks/requests/warehouse/catalog не пагинируются страницами:**
> - `tasks`, `requests` — канбан-доски: фронту нужны все подходящие записи, чтобы считать счётчики по табам/колонкам и группировать (в `tasks/page.tsx` фильтр по цеху делается на клиенте именно ради валидных счётчиков).
> - `warehouse` — фасетные фильтры (металл/марка/толщина) строятся на клиенте из полного набора; постраничная выдача оставит в выпадашках значения только текущей страницы.
> - `catalog` — reference-данные (селекты, дерево категорий, калькулятор); потребители ждут полный список.
>
> Сейчас они ограничены `SAFETY_TAKE=500` — страховка от разрастания, UX не меняется.

#### Вариант 2 (на будущее): серверный фасетинг + пагинация для досок/reference
Если данных станет много (приближение к 500), перевести эти 4 на серверную модель:
1. **Фасеты/счётчики на сервере.** Добавить в GET режим/эндпоинт, возвращающий агрегаты:
   - tasks/requests: `groupBy({ by: ['status'], _count })` (и по `workshopId`/колонкам) → счётчики табов считаются без выгрузки всех строк.
   - warehouse: `distinct` по `metalType`/`steelGrade`/`thickness` (или `groupBy`) → полный список значений для выпадашек независимо от страницы.
2. **Пагинация самих списков.** Перевести основной запрос на `{ items, total, page, pageSize }` через `getPageParams()/paginated()` (как в clients/offers/invoices). Фильтры (`status`, `priority`, `workshopId`, `search`) уже уходят на сервер — оставить, добавить `page/pageSize`.
3. **Фронт.** Доски: рендерить колонку из серверной страницы + «загрузить ещё»/виртуализация; счётчики табов брать из агрегатов (шаг 1), а не из длины массива. Warehouse/catalog: опции фильтров — из фасет-эндпоинта (шаг 1), строки таблицы — из пагинированного списка.
4. **catalog для калькулятора/селектов** оставить отдельным «полным» режимом (`?all=1`) или кэшировать на клиенте — там пагинация не нужна, нужен весь справочник.
5. Поднять/убрать `SAFETY_TAKE` для пагинируемого пути; для «полного» режима catalog оставить явный разумный лимит.
