# План реализации «Личного кабинета клиента»

> Изолированный контур внутри основной CRM с ограниченным функционалом.
>
> **Ключевые решения:**
> - Кабинет = `Client (type=COMPANY)` с флагом портала.
> - Заявки клиента — **отдельная сущность `PortalRequest`**, живут только в кабинете и НЕ попадают в основную CRM автоматически.
> - Перенос в основную CRM — **ручной** (вариант А): админ/менеджер читает портальную заявку и создаёт обычную заявку руками «как надо».
> - Портальные заявки видят **админ (все)** и **ответственный менеджер компании (только своих)**.
> - Один пользователь на компанию.
> - Клиент видит только общий статус, может писать комментарии и прикреплять файлы.

## Границы этапов

**MVP за 15 000 ₽ (первый этап):**
- Схема + миграция + роль `CLIENT`.
- Изоляция/ACL (фундамент безопасности) — обязательно полностью.
- Раздел «Компании» у админа/менеджера: создание кабинета + один пользователь + выбор менеджера.
- Кабинет `/portal`: список своих заявок, создание заявки, своя номенклатура, общий статус, комментарии + файлы.
- Уведомление админу + менеджеру о новой заявке.

**Следующие этапы (отдельно):**
- Кнопка-помощник «Создать заявку в CRM из портальной» (вариант Б).
- Управление несколькими сотрудниками внутри кабинета.
- Уведомления о смене статуса и новых комментариях.
- Право менеджера компании заводить сотрудников.

## 1. Изменения схемы БД (Prisma)

**Новая роль** в enum `UserRole`:
- `CLIENT` — пользователь-клиент (сотрудник внешней компании).

**Модель `User`** — привязка к компании:
- `companyId String?` + связь `company Client? @relation("CompanyUsers", fields:[companyId], references:[id])`.
- Заполнено только у пользователей с ролью `CLIENT`.

**Модель `Client`** — ответственный менеджер и признак «кабинета»:
- `managerId String?` + `manager User? @relation("ClientManager")` — внутренний менеджер CRM (видит заявки этой компании, получает уведомления).
- `isPortalEnabled Boolean @default(false)` — флаг «у компании есть кабинет».
- обратные связи `portalUsers User[]`, `portalRequests PortalRequest[]`, `clientPositions ClientPosition[]`.

**Новая модель `PortalRequest`** — заявка из кабинета (изолирована от `Request`):
- `id, number (autoincrement), companyId (FK Client, Cascade), createdByUserId, title, description, status PortalRequestStatus @default(NEW), createdAt, updatedAt`.
- `items PortalRequestItem[]`, `comments PortalComment[]`, `files PortalFile[]`.
- Нет цен закупки, маржи, производственных подстатусов — их там просто не существует.

**`PortalRequestItem`**: `id, portalRequestId (Cascade), name, quantity, unit`.

**`PortalComment`**: `id, portalRequestId (Cascade), userId, text, createdAt` — общая лента (клиент + внутренние видят одно).

**`PortalFile`**: `id, portalRequestId (Cascade), filename, originalName, size, mimeType, uploadedById, createdAt` — по образцу `RequestFile`, тот же сторадж (`lib/s3.ts`/`storage.ts`).

**`ClientPosition`** — изолированная номенклатура (п.4):
- `id, companyId (FK Client, Cascade), name, unit @default("шт"), createdAt`.

**Enum `PortalRequestStatus`**: `NEW, IN_PROGRESS, READY` (общий статус для клиента; маппинг на основной флоу — на стороне админа при ручном переносе).

**`NotificationType`**: добавить `PORTAL_REQUEST_CREATED`.

Миграция: одна `add_client_portal` (новые таблицы + nullable-поля — безопасно для прода).

## 2. Авторизация и изоляция (критично)

- **Слой 1 — `middleware.ts`:** если `role === "CLIENT"` и путь не начинается на `/portal` → редирект на `/portal`. Добавить `/portal/:path*` и `/companies/:path*` в matcher.
- **Слой 2 — layout** (`app/(dashboard)/layout.tsx`): роль `CLIENT` сюда не пускаем — у неё свой корневой раздел `/portal`.
- **Слой 3 — серверные проверки в каждом API** (главное):
  - Хелпер в `lib/acl.ts`: `getPortalScope(session)` → `companyId` или `403`.
  - Все `/api/portal/*` эндпоинты: жёстко `where.companyId = session.companyId`, игнорируя любой companyId из запроса.
  - `/api/companies/*`: доступ только ADMIN/MANAGER; менеджер видит только компании, где `managerId === его id` (админ — все).
  - `CLIENT` не имеет доступа к существующим CRM-эндпоинтам (`requests, offers, invoices, clients, analytics, warehouse, calculator, catalog, users, tasks, settings`).
- **Слой 4:** портальные DTO в принципе не содержат цен/маржи/произв. статусов — изоляция «по построению».

## 3. Раздел «Компании» (ADMIN + MANAGER)

- **Сайдбар** (`components/layout/DashboardShell.tsx`): пункт `Компании` (только ADMIN/MANAGER).
- **Страницы:**
  - `/companies` — список кабинетов. Админ видит все; менеджер — где он `managerId`. Кнопка «Создать кабинет» — только админ.
  - `/companies/new` — создание: название + выбор менеджера + данные одного пользователя (логин/пароль).
  - `/companies/[id]` — карточка: данные компании, пользователь, вкладки **Заявки** (портальные, переход в просмотр) и **Номенклатура**.
  - `/companies/[id]/requests/[reqId]` — просмотр портальной заявки (состав, комментарии, файлы) для ручного переноса в CRM.
- **API:** `app/api/companies/` — ADMIN/MANAGER, с фильтром по `managerId` для менеджера.

## 4. Кабинет клиента `/portal`

Отдельный корневой layout (как `app/operator/`), меню: Мои заявки, Новая заявка, Моя номенклатура, профиль/выход.

- `/portal` → список своих заявок (номер, название, общий статус, дата).
- `/portal/requests/[id]` → состав, общий статус, комментарии (пишет), файлы (прикрепляет).
- `/portal/requests/new` → форма: контрагента нет (подставляется его компания на сервере), позиции из `ClientPosition` (можно добавить новую прямо здесь).
- `/portal/positions` → CRUD своей номенклатуры.
- **API:** `app/api/portal/*` — сервер всегда берёт `companyId` из сессии, клиент не может его подменить.

## 5. Уведомления

Через существующий `lib/notify.ts` + Telegram:
- **Новая портальная заявка** → `createNotifications` всем ADMIN + менеджеру компании (`PORTAL_REQUEST_CREATED`) + `sendTelegram`.
- (Следующий этап) смена статуса и новые комментарии.

## 6. Порядок работ (фазы)

1. Схема + миграция + роль (`CLIENT`, `companyId`, `managerId`, `PortalRequest*`, `ClientPosition`, `isPortalEnabled`).
2. Изоляция/ACL (middleware + `getPortalScope` + защита `/api/portal` и `/api/companies`) — фундамент.
3. Раздел «Компании» (ADMIN видит все, MANAGER — свои; создание кабинета и пользователя).
4. Кабинет `/portal`: заявки → создание → номенклатура → комментарии/файлы.
5. Уведомления о новой заявке.
6. Тестовый прогон ролевой модели (попытки клиента/менеджера залезть в чужие данные).

## Решения зафиксированы

1. Перенос в CRM — ручной (вариант А). Кнопка-помощник — следующий этап.
2. Заявки клиента изолированы (`PortalRequest`), в основную CRM автоматически не идут.
3. Портальные заявки видят админ (все) + ответственный менеджер (свои).
4. Один пользователь на компанию; клиент пишет комментарии и прикрепляет файлы.
5. Комментарии — общая лента (без скрытых внутренних) на первом этапе.
